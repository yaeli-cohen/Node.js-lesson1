==================================================
תיעוד עבודה עם AI - פרויקט ניהול בית ספר (Express)
==================================================

קובץ זה מתעד את הפרומפטים (Prompts) המרכזיים שנשלחו ל-AI במהלך פיתוח ה-API, ואת דוגמאות המענה שסייעו בבניית הפרויקט ופתרון התקלות.

--------------------------------------------------
פרומפט 1 : מודולריות ופיצול קבצים ב-Node.js
--------------------------------------------------
[הפרומפט שנשלח]:
"What are the advantages of splitting a Node.js project into multiple files? Give practical examples for beginners."

[דוגמה לתשובה/לוגיקה שהתקבלה]:
- קובץ אחד ענק מקשה על קריאות ותחזוקה.
- פיצול לקבצים מאפשר שימוש חוזר בקוד (Reusability) ועבודה חלקה בצוות ללא התנגשויות ב-Git.
- מודול מיוצא ב-CommonJS בעזרת module.exports ומיובא בעזרת require.

--------------------------------------------------
פרומפט 2: שימוש במערכת המודולים המודרנית (ES Modules)
--------------------------------------------------
[הפרומפט שנשלח]:
"Explain Node.js modules to a beginner. Show how one file exports data and another file imports it using import."

[דוגמה לתשובה/לוגיקה שהתקבלה]:
- כדי להשתמש ב-import ו-export, חובה להוסיף ל-package.json את השורה: "type": "module".
- ב-Node.js, כשמייבאים קובץ מקומי עם import, חייבים לציין את סיומת הקובץ המלאה (למשל: ./math.js), אחרת השרת יזרוק שגיאה ויקרוס.

--------------------------------------------------
פרומפט 3: עיקרון "אחריות אחת לכל קובץ" (Single Responsibility) וחיבור שרת
--------------------------------------------------
[הפרומפט שנשלח]:
"What does \"one responsibility per file\" mean in software development? What is usually the responsibility of app.js?"

[דוגמה לתשובה/לוגיקה שהתקבלה]:
- קובץ app.js צריך לשמש רק כ"מרכז בקרה" – הגדרת שרת, Middleware גלובליים (כמו express.json) וחיבור ראוטרים ראשיים.
- כל הלוגיקה והנתיבים הספציפיים חייבים לצאת החוצה לקבצים ייעודיים.
- מומלץ להפריד את app.js מקובץ ההאזנה לפורט (server.js) לצורך בדיקות (Testing) קלות יותר בעתיד.

--------------------------------------------------
פרומפט 4: ארכיטקטורת 4 השכבות (נתיב, קונטרולר, שירות, בסיס נתונים)
--------------------------------------------------
[הפרומפט שנשלח]:
"Describe what happens step by step when a GET request is processed... תראה לי דוגמת קוד מעשית שמחברת את ארבע השכבות"

[דוגמה לתשובה/לוגיקה שהתקבלה]:
מסלול בקשת GET מאובטחת ומחולקת לשכבות:
1. הראוטר (Route Layer) תופס את הכתובת (/api/users/:id).
2. הקונטרולר (Controller Layer) מחלץ את ה-ID ומעביר אותו נקי הלאה.
3. השירות (Service Layer) מפעיל את הלוגיקה העסקית ומסנן שדות רגישים (כמו סינון השדה 'role' לשמירת פרטיות המשתמש).
4. שכבת הנתונים (Data Layer / Repository) מבצעת את השאילתה מול ה-DB בפועל ומחזירה מידע גולמי.


פרומפט 5: שכתוב וארגון מחדש של הפרויקט לפי שכבות (Architecture Refactoring)
--------------------------------------------------
[הפרומפט שנשלח]:
"בשיעור זה יש לשכתב את הפרויקט מחדש, בחלוקה נכונה לתיקיות שונות כמו שנלמד בשיעור: routes, controllers, services, data. הקוד שלי משיעור שעבר: [כאן צורף הקוד המקורי המונוליטי]."

[דוגמה לתשובה/לוגיקה שהתקבלה]:
ה-AI יצר מבנה תיקיות ארכיטקטוני מסודר (Clean Architecture) ופירק את הקובץ היחיד ל-4 שכבות נפרדות לכל אחת מהישויות (Courses, Students):
1. data/ - מערכי הדמה בלבד.
2. services/ - הלוגיקה העסקית והמניפולציה על המערכים.
3. controllers/ - ניהול בקשות ותגובות HTTP (req, res).
4. routes/ - הגדרת ה-Endpoints וקישורם לקונטרולר.
בנוסף, נוצר קובץ app.js רזה המשמש כנקודת כניסה בלבד.

--------------------------------------------------
פרומפט 6: אבטחה לוגית וטיפול שגיאות גלובלי (Code Review & Logical Security)
--------------------------------------------------
[הפרומפט שנשלח]:
"איזה שגיאות עשויות להיות בקוד? איך אפשר לשפר? הער לי הערות על מבנה הקוד? האם נתקן את פונקציית ה-update של הרישומים כדי שתוודא שהקורס והסטודנט קיימים גם בעדכון?"

[דוגמה לתשובה/לוגיקה שהתקבלה]:
ה-AI ביצע קוד ריוויו (Code Review) והצביע על 3 נקודות מרכזיות:
1. פרצה לוגית בעדכון רישומים (אי-בדיקת קיום סטודנט/קורס בזמן Update).
2. חוסר בטיפול שגיאות גלובלי (במקרה של קריסת שרת לא צפויה).
3. אחידות בשמות תיקיות (שינוי מ-service ל-services).
בעקבות זאת ה-AI סיפק קוד מתוקן לשירות ולקונטרולר של ה-enrollments שמבצע בדיקה קפדנית, והוסיף שכבת Middleware גלובלית לטיפול בשגיאות (Error Handling Middleware) ב-app.js שמחזירה תגובת JSON אחידה במקרה של תקלה.
